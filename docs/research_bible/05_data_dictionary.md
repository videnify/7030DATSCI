## Row-count sanity cross-checks (for anyone re-running the pipeline)

- `spy_ohlcv.parquet` and `vix.parquet` each have 2,765 trading-day rows. `spy_returns.parquet` has 2,743 after return/rolling/forward-target warm-up and terminal rows are removed.
- `events_tagged.parquet` has 1,005 rows: 916 economically pre-filtered APP documents + 89 FOMC decisions. GDELT is not included in this sum by design.
- `daily_sentiment.parquet` has 739 event-days and 18 columns. Its explicit health/labour/other counts sum to 14/103/427, exactly matching the catalogue; `gdelt_daily_risk.parquet` independently reproduces the full 4,018-day series.
- `master_dataset.parquet` (2,765 rows, train 2,014 / test 751) has 1 more row per split than `model_features.parquet` (2,013 train / 750 test) — expected, since `master_dataset.parquet` has no lag/rolling-window warm-up trimming (see `dataset_version.md`).
- `feature_matrix.parquet` (2,477 rows, train 1,727 / test 750) has 288 fewer rows than Dataset v1.2, all in the train split. The unchanged 750-row test split confirms that trimming is confined to early warm-up/pre-first-CAR rows. This is deliberate SAP-compliant trimming, not missing-data back-filling.
