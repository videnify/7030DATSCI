## Row-count sanity cross-checks (for anyone re-running the pipeline)

- `prices.parquet`, `spy_ohlcv.parquet`, `vix.parquet` all have 2,765 rows — consistent (same trading-day calendar, 2015-01-01 to 2025-12-31).
- `model_features.parquet` has 2,763 rows — 2 fewer than the raw price series, consistent with the lag/rolling-window features dropping the first few rows via `NaN`.
- `events_tagged.parquet` (180,594) vs. `app_presidential_documents.parquet` (11,629) — the ~15x expansion reflects one-to-many tagging (a single document can be classified into multiple event-type/sentiment-method combinations); confirm this expansion factor is intentional if re-deriving, not a join bug.
- `master_dataset.parquet` (2,765 rows, train 2,014 / test 751) has 1 more row per split than `model_features.parquet` (2,013 train / 750 test) — expected, since `master_dataset.parquet` has no lag/rolling-window warm-up trimming (see `dataset_version.md`).
- `feature_matrix.parquet` (2,511 rows, train 1,761 / test 750) has 254 fewer rows than `master_dataset.parquet`, concentrated entirely in the train split (test row count is identical to `master_dataset.parquet`'s test split, 750, since the longest warm-up window — 200 trading days for `MA200`/`price_vs_ma200` — falls entirely within 2015–2016, nowhere near the 2023-01-03 train/test boundary). This is a larger trim than `model_features.parquet`'s (2 rows) because FES v1.0 enforces `statistical_analysis_plan.md`'s "trim, never back-fill" policy strictly across every engineered column, including the 200-day technical-indicator warm-up and the pre-first-event span of `days_since_car_event` — `model_features.parquet` (built before SAP v1.0 was frozen) did not trim as strictly. This is a deliberate SAP-compliance improvement, not a defect — see `10_decision_log.md`.
