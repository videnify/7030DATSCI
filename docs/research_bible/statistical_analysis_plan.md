## Status of this freeze — read before using this document

This SAP is being frozen **after** Phases 1–8 have already produced results for RQ1 and RQ2 (see `09_results_log.md`, `14_project_dashboard.md`). A clinical-trial SAP is normally frozen *before* any data is analysed; that is not literally true here, and pretending otherwise would be a form of HARKing (Hypothesising After Results are Known — the exact risk `02_hypotheses.md` already warns against for hypothesis changes). This document handles that honestly rather than silently:

1. **RQ1 and RQ2 methodology is ratified, not newly invented.** Every test, threshold and policy below was frozen before the final reporting pass. On 2026-07-15 the already-prespecified BH-FDR correction, 95% event-type mean intervals and Cohen's d table were implemented without changing the underlying CAR observations or the SAP decision rule; `rq1_reporting_validation.json` records `PASS`.
2. **RQ3 is genuinely prospective.** No baseline model exists yet (`07_model_plan.md`); the model-comparison tests (Diebold–Mariano, two-proportion z-test, Bonferroni correction) below are frozen *before* that analysis runs. This is the part of the SAP that behaves exactly like a pre-registered trial protocol, and is the part that most matters going forward (Missions 05–08).
3. **Anything below marked "🆕 New in v1.0"** is a policy this project did not previously have written down anywhere (e.g. outlier policy, stationarity policy, VIF/correlation/variance thresholds, prediction intervals). These are frozen now, before Mission 05 (Feature Matrix v1.0) touches them, so they genuinely precede their first use — this is where "Statistical Design Freeze" does its intended job.

If a supervisor or examiner asks "was this pre-registered," the honest answer is: partially — RQ3's remaining protocol is; RQ1/RQ2's already-run methodology is formalised in the same document for completeness and audit clarity, and that distinction is stated here rather than hidden.

## PART A — Global Statistical Policy

Every policy below states **what** is frozen and **why**, per the mission brief's requirement that no policy exist without a stated reason.

| Policy | Frozen value | Why |
|---|---|---|
| Significance level (α) | **0.05** | Already the threshold used throughout `02_hypotheses.md`, `04_statistics_plan.md`, and `09_results_log.md` for RQ1/RQ2; changing it now would silently invalidate results already reported. 0.05 is also the conventional threshold in event-study and applied-finance literature this project sits within (Fama, Fisher, Jensen & Roll, 1969 tradition). |
| Confidence level | **95%** | Complement of α = 0.05; used consistently for every CI in this project (DoWhy backdoor CI, planned RMSE-difference bootstrap CI). |
| One-sided vs. two-sided testing | **Two-sided by default.** One-sided permitted only for the RQ3 model-vs-baseline comparisons (H3), where the hypothesis is explicitly directional ("full-feature model outperforms baseline," not "differs from") — see `02_hypotheses.md` H3 and `statistical_decision_matrix.md` Part K. | RQ1 (does an event move price, in either direction) and RQ2 (is a feature important) have no pre-specified direction — a two-sided test is the conservative, defensible default. RQ3's decision rule was already written directionally ("outperform") before this freeze, so one-sided testing there is a formalisation of existing intent, not a new choice made to inflate significance — it is logged explicitly so it cannot be mistaken for p-hacking. |
| Multiple comparison policy | Benjamini-Hochberg FDR (RQ1, 5 event-type tests); Bonferroni α/3 = 0.0167 (RQ3, 3 models vs. baseline); no correction for RQ2 (descriptive ranking, not a test family) | Matches the family sizes and test purposes exactly as already specified in `04_statistics_plan.md` — restated here as frozen, not re-derived. See `statistical_decision_matrix.md` for the full family table. |
| Missing data policy | Forward-fill for lower-frequency macro series onto the trading calendar; zero-fill for event/sentiment aggregates (absence of a tagged event is a true zero, not an unknown); no imputation of exchange-traded price/VIX series (a gap there is a collection defect, not a modelling problem); lag/rolling-window warm-up rows are **trimmed**, never back-filled or interpolated | Matches `dataset_version.md`'s already-implemented and validated policy exactly — restated here as the frozen project-wide rule so Mission 05 (feature matrix rebuild) does not silently deviate. Imputing a warm-up row would fabricate pre-history signal that never existed. |
| Outlier policy 🆕 | **No outlier removal or winsorisation is applied to returns, CAR, or sentiment scores.** Extreme values (e.g. COVID-19 crash daily returns) are retained as-is. | In event-driven market research, "outliers" in daily returns are frequently the exact phenomenon under study (a large move coinciding with an event *is* the signal, not noise contaminating it). Removing them would bias RQ1 toward finding smaller, safer effects and would be indefensible without a data-quality (not a statistical) justification. Heavy tails are instead handled by preferring non-parametric/robust tests over Gaussian ones wherever residual diagnostics show non-normality (already the case for LASSO residuals — kurtosis 17.26, `04_statistics_plan.md`). |
| Transformation policy 🆕 | Log returns (`log(P_t / P_{t-1})`) are the only transformation applied to price series; no Box-Cox or other variance-stabilising transform is applied on top. Macro/sentiment/VIX features are used in raw, differenced, or moving-average-relative form as engineered (`06_feature_dictionary.md`), never log-transformed (VIX and macro levels are not price series and log-transforming them would not carry the same "return" interpretation). | Log returns are the standard, already-adopted convention in this project (`master_dataset.parquet: log_return`) and in the event-study literature it follows. Freezing "no further transform" prevents a later notebook from quietly re-transforming a variable and breaking comparability with already-reported numbers. |
| Feature scaling policy | `StandardScaler`, fit on the training split only, applied to the full feature matrix uniformly across all three candidate models (including tree models that do not strictly require scaling) | Matches `03_methodology.md` §5 exactly. Scaling every model identically (even where technically unnecessary for XGBoost/LightGBM) keeps the input matrix identical across models, isolating model architecture as the only varying factor in the RQ3 comparison — a scaling asymmetry between models would confound the comparison. |
| Stationarity policy 🆕 | SPY log returns, VIX, and each macro series must be tested for stationarity (ADF + KPSS, see `statistical_assumptions.md` Part H) in `03_exploratory_analysis.ipynb` before any feature built from them is treated as safe for a linear model (LASSO) that assumes a stable data-generating process. Log returns are expected to pass (return series are the standard stationarity-inducing transform of a price level); if a macro **level** series (e.g. `cpi`, `fed_funds_rate`) fails, its **first difference** (already engineered as `cpi_mom`, `fed_rate_change`) is the version used in modelling, not the raw level. | Neither ADF nor KPSS has been run in this project yet — a genuine gap this SAP closes prospectively, before Mission 05 rebuilds the feature matrix. Tree-based models (XGBoost/LightGBM) are not strictly assumption-bound here, but LASSO is, and all three models share one feature matrix, so the check applies project-wide. |
| Model selection policy | Primary model-ranking metric: **test-set RMSE**, tie-broken by directional accuracy. An event-informed model is only reported as "beating the baseline" if it wins on **both** RMSE (Diebold-Mariano) **and** directional accuracy (two-proportion z-test) at the corrected α — a model winning on one metric only is reported as a mixed result, never rounded up to "outperforms." | Restates `07_model_plan.md`'s already-specified rule as a global, frozen policy so it cannot be loosened later if a model narrowly misses on one metric but the temptation exists to report it as a win anyway. |
| Reporting policy | Every reported "significant" result states: test used, test statistic, exact p-value (not just "p<0.05" unless p<0.001), sample size, effect size, 95% CI (where applicable), and the multiple-comparison correction applied | Restates `04_statistics_plan.md` "General conventions," extended with the exact-p-value and effect-size requirements made explicit and mandatory in `statistical_reporting_guidelines.md`. |
| Confidence interval policy | A 95% CI is reported alongside every point estimate that feeds an RQ1–RQ3 claim. Event-type mean-CAR t intervals, DoWhy CIs, empirical prediction intervals, seeded block-bootstrap RMSE intervals and Wilson directional-accuracy intervals are implemented. | A point estimate without an interval invites overclaiming precision, especially with far fewer independent events than trading days. |
| Effect size policy | Event-type t-tests report Cohen's d alongside raw p and BH q; causal estimates report the point estimate in native return units (current pooled estimate +0.005601); RQ3 comparisons report % RMSE and percentage-point directional-accuracy differences | Statistical significance alone is insufficient; the completed event-type table shows only negligible-to-small magnitudes. |
| P-value interpretation policy | A p-value is the probability of observing a test statistic this extreme *if H0 were true* — never interpreted as "the probability H0 is true" or "the probability the result is real." No test in this project is added, dropped, or re-run after seeing its p-value (all RQ3 tests are pre-specified in `statistical_decision_matrix.md` before the baseline model is trained). | Standard statistical-literacy discipline; explicitly stated because a dissertation examiner will check for this exact misinterpretation in the write-up. |
| Reproducibility policy | `config.yaml: model.random_seed = 42` fixes all randomness project-wide; `TimeSeriesSplit`, never a random/shuffled split; every processed artefact is derived and versioned (`dataset_version.md`), never hand-edited | Restates `03_methodology.md`'s existing reproducibility notes as a frozen, dissertation-citable policy. |
