# Statistical Assumption Matrix (SAP v1.0)

**Purpose:** Every statistical assumption relied on anywhere in the pipeline — why it matters, how it is checked, which notebook checks it, how to interpret the check, and what to do if it fails. Written so a violated assumption is never silently ignored, and so "we checked" is always backed by a named test rather than an eyeballed plot.
**Owner:** Research Statistician sign-off.
**Dependencies:** `statistical_analysis_plan.md` Part A (policies this matrix operationalises), `statistical_decision_matrix.md` (the tests this matrix's checks gate).
**Update Frequency:** Add a row whenever a new statistical method (and therefore a new assumption) is introduced — must accompany a Version 2+ SAP amendment.
**Relation to Dissertation:** Direct source for the assumption-checking sub-sections of Chapter 3 §3.4 and the diagnostic commentary in Chapter 4/5.

## Assumption matrix

| Assumption | Why it matters here | How tested | Notebook | Interpretation | Action if violated |
|---|---|---|---|---|---|
| **Normality** (of CAR within event type; of model residuals) | The event-study t-test and any Gaussian-CI claim formally assumes approximately normal errors; financial returns are well known to be heavy-tailed | Skewness/kurtosis inspection + QQ-plot (CAR, `03_exploratory_analysis.ipynb`); Jarque-Bera test (LASSO residuals, already run — p≈0.0, kurtosis 17.26) | `03_exploratory_analysis.ipynb` (CAR), `07_model_evaluation.ipynb` (residuals) | Non-normality is the **expected**, literature-consistent result for daily financial returns, not a modelling error | Prefer the non-parametric/robust alternative already named in `statistical_decision_matrix.md` (CAAR aggregation before testing rather than pooling raw CAR; bootstrap CI over Gaussian CI for RMSE differences) — never proceed with a parametric p-value as if the check had passed |
| **Stationarity** 🆕 | A linear model (LASSO) implicitly assumes a stable relationship between features and target; a non-stationary regressor (e.g. a macro level series with a trend) can produce spurious in-sample fit that collapses out-of-sample | ADF test (H0: unit root) + KPSS test (H0: stationary) — run both since they have opposite nulls and neither alone is conclusive | `03_exploratory_analysis.ipynb` | Log returns are expected to pass (stationarity-inducing transform already applied); macro **levels** (`cpi`, `fed_funds_rate`, `treasury_10y`) are expected to fail and are not used directly — their engineered first-differences (`cpi_mom`, `fed_rate_change`, `yield_spread_change`) are used instead | If a feature already treated as "safe" (e.g. a rolling/technical indicator) fails both tests, exclude it from LASSO or difference it before use; tree models are less sensitive to this but the shared feature matrix means the check applies project-wide |
| **Linearity** | LASSO assumes a linear feature-target relationship; DoWhy's `backdoor.linear_regression` estimator assumes linearity in the adjustment formula | Residual-vs-fitted plot; partial-dependence comparison between LASSO coefficients and SHAP dependence shape for the same feature (nonlinear SHAP shape flags a linearity violation) | `07_model_evaluation.ipynb` | A feature whose SHAP dependence plot is strongly non-monotonic/non-linear but is used linearly in LASSO/DoWhy is a candidate under-fit signal for that specific feature | Report the discrepancy explicitly in Discussion §5.2 (do not silently prefer whichever model's answer is more convenient); tree-based models (XGBoost/LightGBM) are not linearity-constrained and serve as the non-parametric cross-check already built into the model set |
| **Independence** | The event-study t-test assumes independent draws; presidential communications cluster in time (multiple statements in one crisis week) | Visual timeline inspection of event dates (`02_eda.ipynb` figures `02g`–`02l`); mitigated methodologically (CAAR aggregation, not raw CAR pooling) rather than tested directly, since a formal independence test for irregularly-spaced event clustering is not standard practice here | `02_eda.ipynb`, `04_causal_analysis.ipynb` | Known, acknowledged violation (`11_limitations.md` L4) — not something a single test either "passes" or "fails," but a structural property of unstructured political communication data | Test on CAAR (event-type averages) rather than pooled raw CAR — already the adopted mitigation; state as a literature-acknowledged limitation in the write-up, not as an unaddressed error |
| **Autocorrelation** (of model residuals) | Autocorrelated residuals indicate the model has not captured all temporal structure, and would bias standard errors used for any residual-based inference | Durbin-Watson statistic (LASSO residuals, already run — 2.02, no material autocorrelation) | `07_model_evaluation.ipynb` | DW ≈ 2 indicates no material autocorrelation — the current LASSO result passes | If a future re-tuned model shows DW materially away from 2, investigate omitted lag structure before trusting its residual-based diagnostics |
| **Multicollinearity** 🆕 | LASSO's L1 penalty handles correlated predictors by shrinkage (not by failing), but coefficient *interpretation* becomes unreliable under severe collinearity, and interaction features are, by construction, correlated with their parent features | Variance Inflation Factor (VIF), threshold 10; pairwise Pearson correlation, threshold \|r\| > 0.90 (both frozen in `statistical_decision_matrix.md`) | `05_feature_engineering.ipynb` (to be added, Mission 05) | A VIF > 10 or a correlated pair flags a feature whose individual coefficient/importance should be interpreted cautiously, not automatically removed — RF/SHAP importance is comparatively robust to this, so RQ2's ranking claim is not undermined by a VIF flag on an individual feature | Document flagged features in `06_feature_dictionary.md`; only drop a feature if it is both highly collinear **and** redundant in RF importance terms (near-duplicate signal, not just correlated) — collinearity alone is not sufficient grounds for removal given LASSO's regularisation already manages it |
| **Homoscedasticity** | Heteroskedastic residuals (common in financial return series — volatility clustering) bias standard-error-based inference if unaddressed | Breusch-Pagan-style heteroskedasticity correlation statistic (LASSO residuals, already run — 0.199, mild) | `07_model_evaluation.ipynb` | Mild heteroskedasticity present but not severe | Already the reason this project prefers bootstrap/non-parametric CIs over standard-error-based Gaussian CIs for the RQ3 model comparisons (`statistical_analysis_plan.md` Part A, CI policy) |
| **Temporal leakage** | The single most consequential assumption in this project — any feature that indirectly encodes future information would invalidate every RQ2/RQ3 result | `master_dataset.parquet` validation report: `fwd_return_1d[t] == log_return[t+1]` exact-match check (0 mismatches), last-row `NaN` proof, "no centred/backward window reaching into t+1" rule (`dataset_version.md`) | `01_data_collection.ipynb` (raw), dataset-freeze validation (Mission 03) | Zero mismatches and a NaN final row are the two mechanical proofs that no future value has been folded into a feature; this must be re-verified after Mission 05 rebuilds the feature matrix, not assumed to still hold | Any new lag/rolling feature must use `.shift(n)` with `n ≥ 0` relative to `date` — this rule is a hard gate, not a guideline; Mission 05 must re-run the leakage check on the rebuilt `model_features.parquet`, not only on `master_dataset.parquet` |
| **Train/Test independence** | A chronological split can still leak information if rolling-window features computed *before* the split boundary reach across it (e.g. a 21-day rolling feature on the first test-set row partially reflects train-period data) | Confirm the split boundary date and the longest rolling-window length (63 days, `momentum_63d`) — the first ~63 test-set rows' rolling features partially span the train period by construction; this is a known, accepted property of rolling-window features (not leakage of the *target*, only of *past feature history*, which is legitimate for a forecasting task) | `05_feature_engineering.ipynb` | This is **not** target leakage (the target is never touched) — it is the ordinary and correct behaviour of a rolling feature at a split boundary, and does not violate the chronological-split principle | State explicitly in Methods §3.3 so an examiner does not mistake this for a leakage bug; do not "fix" it by dropping the first 63 test rows unless a supervisor specifically requests a stricter boundary buffer |

## PART H — Time Series Diagnostics (frozen specification)

| Diagnostic | Definition / why it exists | Applied to | Lag/window choice and why |
|---|---|---|---|
| **ADF** (Augmented Dickey-Fuller) | Tests H0: series has a unit root (non-stationary). Chosen as the primary stationarity test because it is the standard first check in applied financial time-series work and directly motivates the "use log returns, not price levels" convention already adopted | SPY log return, VIX, each macro series (level and first-differenced version) | Standard automatic lag-length selection (AIC-minimising), not hand-tuned — avoids an arbitrary lag choice biasing the test |
| **KPSS** | Tests H0: series is stationary (opposite null to ADF) — run as a cross-check because ADF alone has known low power against near-unit-root alternatives | Same series as ADF | Same automatic lag selection convention |
| **ACF** (Autocorrelation Function) | Visualises serial dependence structure — used to justify which lagged-return features (`return_lag1d/3d/5d/10d/21d`) are worth engineering, and to sanity-check that lag choice isn't arbitrary | SPY log return | Displayed to 30 lags (~6 weeks) — long enough to see any monthly seasonality, short enough to stay readable |
| **PACF** (Partial ACF) | Isolates the direct effect of each lag, controlling for shorter lags — used to confirm the specific lag set already engineered (1/3/5/10/21 days) captures the dominant direct dependence rather than being an arbitrary round-number choice | SPY log return | Same 30-lag display window as ACF |
| **Rolling statistics** | Rolling mean/std of returns and VIX visualise regime change (volatility clustering) that motivates the `vix_regime`, `volatility_10d/21d/63d` engineered features | SPY log return, VIX | Windows of 10/21/63 trading days — matches the windows already chosen for the technical-indicator feature set (`06_feature_dictionary.md`), so the diagnostic and the feature engineering use the same, consistent window definitions rather than introducing a fourth ad hoc window |
| **Lag selection** | The 1/3/5/10/21-day lag set is not arbitrary: 1 day (immediate autocorrelation), 3/5 days (within-week), 10 days (two-week), 21 days (~1 trading month) — chosen to span the horizons a next-day return predictor could plausibly exploit without engineering redundant adjacent lags (e.g. 1–2–3–4 days, which would mostly duplicate signal) | All lagged price/return features | Cross-referenced against ACF/PACF once run (Mission 05) — if ACF/PACF show negligible dependence beyond a shorter horizon, that is reported as a finding (motivating the existing feature set), not a reason to silently add more lags post hoc |
